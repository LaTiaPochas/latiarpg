# LaTiaRPG - Base Web RPG

Proyecto base para RPG web usando:

- [Next.js](https://nextjs.org/)
- [Supabase](https://supabase.com/) (Base de datos + auth opcional)
- [Vercel](https://vercel.com/) (deploy)

## 1) Ejecutar en local

```bash
npm install
copy .env.example .env.local
```

Completa `.env.local` con tus credenciales de Supabase:

```bash
NEXT_PUBLIC_SUPABASE_URL=...
NEXT_PUBLIC_SUPABASE_ANON_KEY=...
```

Luego:

```bash
npm run dev
```

Abre [http://localhost:3000](http://localhost:3000)

## 2) Configurar Supabase

1. Crea proyecto en [Supabase Dashboard](https://supabase.com/dashboard).
2. Ve a **Project Settings -> API**.
3. Copia:
   - `Project URL` -> `NEXT_PUBLIC_SUPABASE_URL`
   - `anon public key` -> `NEXT_PUBLIC_SUPABASE_ANON_KEY`
4. En **SQL Editor**, crea una tabla de ejemplo:

```sql
create table if not exists public.rpg_classes (
  id bigint generated by default as identity primary key,
  name text not null,
  description text
);

insert into public.rpg_classes (name, description)
values
  ('Guerrero', 'Alta vida y defensa.'),
  ('Mago', 'Alto dano magico, baja defensa.'),
  ('Arquero', 'Ataques a distancia y criticos.');
```

Con eso, la home ya mostrara clases desde Supabase.

## 3) Subir al repo GitHub

Repo objetivo: [LaTiaPochas/latiarpg](https://github.com/LaTiaPochas/latiarpg.git)

Si aun no lo conectaste como remoto:

```bash
git remote add origin https://github.com/LaTiaPochas/latiarpg.git
git branch -M main
git add .
git commit -m "chore: bootstrap Next.js RPG with Supabase integration"
git push -u origin main
```

## 4) Deploy en Vercel

1. Entra a [Vercel](https://vercel.com/new).
2. Importa el repo `latiarpg`.
3. Framework detectado: **Next.js**.
4. En **Environment Variables**, agrega:
   - `NEXT_PUBLIC_SUPABASE_URL`
   - `NEXT_PUBLIC_SUPABASE_ANON_KEY`
5. Deploy.

Cada push a `main` disparara un redeploy automatico.

## 5) Siguientes pasos recomendados para tu RPG

1. **Auth** de Supabase (email/google).
2. Tabla `profiles` para datos del jugador.
3. Tablas `characters`, `inventory`, `quests`, `combat_logs`.
4. Row Level Security (RLS) para aislar datos por usuario.
5. Mapa/combate por turnos en UI (estado del cliente + guardado server).
